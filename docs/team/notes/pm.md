# PM 누적 노트

> PM이 매 미션에서 발견한 패턴·인사이트·차별화 노하우를 누적하는 노트.
> 호출 시 작업 시작 전에 본 파일을 먼저 읽고 이전 학습을 회복한다.

## 차별화 5축 (calc-tools 정체성)
1. **한국 특화도 100%** — 글로벌 계산기가 못 만드는 한국 노동법·세법 도구
2. **법령 1차 출처** — 법제처 직링크 (E-E-A-T 신호)
3. **교육 + 도구 결합** — 계산만 하지 않고 왜·어떻게·얼마인지 끝까지 설명
4. **Grey Zone 영리 활용** — 사용자가 받을 수 있는 합법적 최대 효용 적극 발굴
5. **단일 진실 공급원 (rates.json)** — 매년 변경 수치도 한 곳만 갱신

## 패턴 라이브러리

### 미션 분배 결정 트리
- 가이드 글 작성 → Compliance(본문) → Brand(검수) → DevOps(sitemap) → QA(링크)
- 산식 검증 → QA → Compliance → DevOps → 보고
- rates 갱신 → DevOps → Compliance → QA
- UI 변경 → Brand → DevOps → QA
- AdSense 점검 → 전원 (DevOps + Compliance + Brand + QA + PM 종합)

### 인사이트
- 가이드 글 템플릿이 한 번 정립되면 작성 속도 4~5배 (세션 39)
- 글 1편당 800~1500단어가 적정 (한 세션 5~7편)
- 산식 검증 패스는 작성 직후가 아니라 글 모두 완성 후 별도로 (산식 오류는 1번 패스로 못 잡음)

## AdSense 4차 점검 패턴 (2026-04-12 신규)

### 숫자 기반 GO/NO-GO 판정 원칙
"괜찮아 보입니다" 절대 금지. 다음 5개 축을 모두 숫자로 측정한다.

| 축 | 측정 방법 | 통과선 |
|---|---|---|
| 콘텐츠 볼륨 | `sed 's/<[^>]*>//g' FILE | wc -w` | 핵심 도구 1,800+ / 가이드 800+ |
| 광고 밀도 | `grep -c 'class="ad-space"'` / `grep -c '<ins class="adsbygoogle"'` | 신청 전 페이지당 1 이하 |
| 섹션 깊이 | `grep -cE "<h2"` | 핵심 도구 8+ |
| 1차 출처 | `grep -cE "law.go.kr|nts.go.kr|moel.go.kr|nhis.or.kr|korea.kr"` | 도구당 2+ |
| SEO 무결성 | canonical 1, JSON-LD 1, sitemap 등록, robots 모순 0 | 100% |

### AdSense 4차 통과 치명 함정 3가지 (본 미션에서 실제 발견)
1. **광고 슬롯 미축소** — 본문을 아무리 늘려도 광고 2개면 "Low Value Content" 동일 판정. CLAUDE.md 정책 "페이지당 1개"를 실제 파일에서 `grep -c` 로 확인하지 않으면 누락 확률 100%.
2. **sitemap × robots 모순** — privacy.html을 sitemap에 등록하면서 robots.txt는 Disallow. GSC "Blocked by robots.txt" 경고 = 심사관 부정 신호.
3. **컨셉 전환 후 1일 신청** — 구글 크롤러가 구컨셉 캐시를 여전히 보관 중. 최소 7~10일 대기 + GSC URL 검사 재요청 필수.

### 완벽주의 함정 → Grey Zone 선택
- Phase C 30편 완성 전까지 신청 대기 ≠ 정답
- **20~22편(노동 12 + 세법 5~6 + 금융 3~4)이면 E-E-A-T 신호로 충분**
- 나머지는 통과 후 수익화 저해 없이 작성 가능 → 타이밍이 더 소중

### 광고 복구 전략
- 신청 전: Auto Ads만 (가장 보수적, 옵션 B 권고)
- 통과 후: 핵심 도구 도구당 최대 2개까지 점진 복구
- 3번 거절된 상황에서 "광고 밀도"를 의심 항목 1순위로 가정하고 대응

### 신청 전 필수 체크리스트 (PM 종합 GO 시점에 main Claude 실행 대본)
1. 광고 슬롯 `grep -c 'class="ad-space"'` 핵심8 + 가이드12 + 루트5 모두 ≤1
2. robots.txt `cat` + sitemap.xml `grep privacy` 모순 0
3. `scripts/word-count.sh` 실행 → 핵심 도구 평균 1,800+
4. `scripts/check-stale.sh` 실행 → 에러 0
5. Lighthouse 모바일 핵심 8개 Performance/SEO 90+
6. GSC 색인율 조회 (사용자 수동)
7. `git log --since` 로 3차 거절 후 커밋 수 확인 (실제 개선 증명)
8. ads.txt 내용 확인 `google.com, pub-9632606296515258, DIRECT, f08c47fec0942fa0`
9. OG 태그 구컨셉 잔재 `grep "계산기 모음"` → 0
10. 사용자에게 최종 신청 승인 요청

## 4차 GO/NO-GO 사이클 학습 (2026-04-12 세션3)

### 패턴: "치명 3건 해소 후 조건부 GO"
- NO-GO에서 GO로 전환하는 데 필요했던 것: worker 4명 병렬 투입 1사이클
- 치명 항목은 3개였고 전부 코드/설정 수정으로 해결 가능했음 (시간 blocking 항목은 별도)
- **교훈**: NO-GO 판정 시 "치명 vs 시간 vs 완벽주의" 3분류를 명확히 해야 함. 치명은 즉시 해소, 시간은 대기, 완벽주의는 버림.

### 패턴: "20편 Grey Zone"
- 30편 목표 중 20편이면 E-E-A-T 충분. 나머지 10편은 승인 후.
- 이 판단이 "완벽주의 함정 회피"의 실전 적용 사례. 향후 유사 판단에 재활용.

### 패턴: "컨셉 전환 경과일이 유일한 blocking"
- 코드 완성도와 무관하게 구글 캐시 갱신 시간이 최종 게이트.
- 이런 경우 "조건부 GO"가 정답. GO도 NO-GO도 아닌 중간 판정이 필요한 전형적 사례.

### 통과 확률 추정 방법론
- 3연속 거절 이력 = 기본 -10%p (심사관 편견 가능)
- 치명 해소 = 각 +10%p
- Grey Zone/교육 콘텐츠 = +5~10%p
- 보수 60%, 중립 70%가 현실적 범위

### worker 병렬 투입 효율
- DevOps + Compliance + Brand + QA 4명이 각자 영역을 동시 처리
- 단, 순서 의존성 존재: Compliance가 가이드 작성 → QA가 sitemap 추가 + 링크 검증 / Brand가 연도 잔재 수정
- PM이 미리 의존성 그래프를 board에 명시하면 후속 worker 대기시간 최소화
